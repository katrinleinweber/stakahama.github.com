Vi[m] emulation with viper-mode and vimpulse

I was interested in how my workflow would change if I followed the operational paradigm of vi: 

"You are always in normal mode, and only enter insert mode for short
bursts of typing text, after which you press <Esc> to go to normal
mode."

(Ctrl-[ can also be used instead of Esc).

See also
http://jmcpherson.org/editing.html
http://www.moolenaar.net/habits.html

Viper-mode (standard with Emacs) allows modal editing (in the vi sense) in Emacs, and vimpulse.el extends this to include visual-mode. I set out with these customizations in my .viper file: 

(setq viper-inhibit-startup-message 't)
(setq viper-expert-level 1)

More than the mechanics of it I thought it might affect the way I hammered out ideas. I found out that I was very slow ;).

I have not used vi[m] extensively, but the lack of C-o (switch to normal mode for a single command) in viper/vimpulse may be a drawback for vi[m] users (to my knowledge, it is unimplemented).

Interestingly (and surprisingly), the modal style of editing ('mode' here used in the vi sense) was not a deliberate design decision over the emacs model: 

"One of the good things about EMACS, though, is its programmability
and the modelessness. Those are two ideas which never occurred to
me." - Bill Joy

Conceptually, Emacs can be thought of as being in insert/normal mode by default, and enters command mode while Ctrl is pressed.

Managing processes with emacs server

I have (server-start) in my .emacs file and for most tasks use open files through open instance or use emacsclient (mapped to aliases ec, em, ea). Emacs 23 also provides the option of starting the server with emacs --daemon (which can be loaded on startup) but I do not use it. 

## --- emacsclient ---
## http://philipweaver.blogspot.com/2009/08/emacs-23.html
# start a windowed frame
alias ec="/Applications/Emacs.app/Contents/MacOS/bin/emacsclient -n -c -a /Applications/Emacs.app/Contents/MacOS/Emacs"
# start a terminal frame
alias em="/Applications/Emacs.app/Contents/MacOS/bin/emacsclient -t -a /Applications/Emacs.app/Contents/MacOS/Emacs -nw"
# do not start a new frame
alias ea="/Applications/Emacs.app/Contents/MacOS/bin/emacsclient -n -a /Applications/Emacs.app/Contents/MacOS/Emacs"
## --- kill emacs ---
alias killemacs='emacsclient -e "(kill-emacs)"'

? launching single vs multiple instances

On the occassion that I need more instances, I will fire up another emacs instance through the terminal, which will give an error on OS X: 

$ 2009-09-07 15:33:50.454 Emacs[2164:10b] *** CFMessagePort:
bootstrap_register(): failed 1100 (0x44c) 'Permission denied', port = 0x3803,
name = 'org.gnu.Emacs.ServiceProvider'
See /usr/include/servers/bootstrap_defs.h for the error codes.

But the new instances seem to work even if I ignore this error (Carbon Emacs Wiki warns of warnings). This type of error is documented on the web for other applications, but most people seem to ignore it also.

Spell check

Spell-checking:

  • ispell
  • flyspell (I prefer invoking flyspell-buffer after editing the text rather than using flyspell-mode while typing [it's slow])
  • speck-mode

If writing in LaTeX, sometimes I export the .tex file to .rtf using latex2rtf and open in MS Word.
